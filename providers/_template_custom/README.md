# 自定义流程 Provider 模板

这个模板适合标准 UI 表达不完的平台，例如：

- 需要多步骤登录、保存凭证、权限检测。
- 需要先读取空间、知识库、目录，再继续导出或导入。
- 需要图片补全、失败项重试、断点续跑、特殊报告。
- 平台按钮或流程和普通导入导出差异很大。

## 当前推荐做法

先不要直接改主程序 UI。推荐把复杂流程拆成几个动作：

- 登录 / 保存凭证
- 权限检测
- 读取目录
- 导出或导入
- 失败项重试
- 生成报告

如果标准字段和按钮仍然不够，请在 PR 里说明你需要的专属 UI。维护者会评估是否把它升级成内置 provider 或后续沙箱自定义 UI。

## 为什么不直接注入任意 HTML

万能导是桌面端应用，provider 脚本会在用户本机执行。如果直接允许任意前端代码进入主窗口，容易带来安全和维护问题。

所以当前规则是：

- 简单平台：优先用标准 UI 模板。
- 复杂平台：先提交 Python 核心逻辑、README 和流程说明。
- 需要专属 UI：由维护者评估后接入，确保只影响该平台页面，不影响其他平台。

## PR 中需要写清楚

- 为什么标准 UI 不够。
- 需要哪些特殊步骤或按钮。
- 登录凭证保存在哪里，如何过期提示。
- 是否支持目录结构、图片、附件、批量、断点续跑和失败重试。
- 是否已经做过真实导出或导入测试。
- 是否参考第三方项目，以及对应许可证。

## AI 辅助开发提示词

```text
请参考 docs/插件开发指南.md 和 providers/_template_custom，为某某复杂平台实现万能导 provider。
要求：
1. 先不要改 Electron 主程序，优先把流程拆成 provider.json actions。
2. 如果确实需要专属 UI，请在 README.md 写明 UI 草图、按钮、字段和状态流转。
3. Python 脚本必须能独立运行并输出最终 JSON。
4. 不提交 Cookie、Token、账号密码或私人测试数据。
5. 在 README.md 写明测试结果、限制和失败处理。
```
